## Top 3 Priority Fixes

1. **BLOCK — Remove all fabricated data (ticker + live counter).**
   Impact: destroys trust the moment a creator notices the score for "Canva Business Bundle" is always exactly 82 — the same number, every visit. Violates PBA rule 8 ("과장 카피 금지") and general honesty principles. Fix: replace ticker with static testimonial quotes or remove it; replace live counter with a real Supabase query or remove entirely. Lines 1222-1265.

2. **FLAG — Move the form above the fold or add a sticky CTA.**
   Impact: the primary conversion action (form submit) is reachable only after scrolling past founding-bar → hero → vs-row → stat-row → ticker → vitals → how-it-works. A non-technical Gumroad/Etsy seller will bounce before reaching it. Fix: either (a) collapse hero + form into a single above-fold unit, or (b) add a sticky "Check My Score" button that anchors to the form. Lines 788-929.

3. **FLAG — Widen the mobile breakpoint from 480px to 768px.**
   Impact: on devices 481-767px wide (the majority of modern phones in landscape, small tablets) the 3-column stat-row, vitals grid, and how-steps grid render at desktop density, causing cramped unreadable layouts. Fix: change `@media (max-width: 480px)` to `@media (max-width: 767px)` and test vitals-grid and stat-row at 375px, 414px, and 667px. Line 774.

## Detailed Findings

### Pillar 1: Visual Hierarchy & Typography (2/4)

**Strengths:**
- H1 at `2.25rem / font-weight: 800` (line 85-89) creates a clear dominant entry point.
- Yellow accent (`#FFD700`) is reserved for interactive and key semantic elements — correct usage.
- `Press Start 2P` pixel font used only for labels, buttons, and scores — not body text — appropriate restraint.

**Issues:**

**Typography scale is wide but not always intentional.**
Sizes found across the file:
- `0.35rem` (vital-name, line 599) — renders at ~5.6px on 16px base. Invisible on most displays.
- `0.38rem` (how-num, line 760) — same problem.
- `0.42rem` (qitem-rank, sitem-n, lines 377, 401) — borderline legible at 6.7px.
- `0.45rem` used in 4 separate contexts: hero-eyebrow (77), score-label (290), sec-head (348), loading-label (247).
- `0.48rem` loading-label duplicate variant (line 248).

Five distinct sub-0.5rem sizes are employed for decorative labels. These are functionally invisible below 16px base font on most browsers. They communicate aesthetic intent but deliver zero readability to target audience (non-technical creators on phones).

**Muted colors on critical helper text are too low-contrast.**
- `.vital-desc` at `color: #444` (line 606) on `#0d0d0d` background: estimated contrast ratio ~2.3:1, failing WCAG AA (4.5:1 minimum for small text).
- `.how-desc` at `color: #444` (line 767): same issue.
- `.loading-step` at `color: #333` (line 255): even worse, ~1.9:1.
- `.stat-label` at `color: #444` (line 669): fails.

These are not decorative — they carry explanatory content for the target user.

**Hierarchy between sections is undifferentiated.**
Every content block (vs-row, stat-row, ticker, vitals, how-it-works, form) has similar visual weight and spacing (~1.75-2rem margin-bottom). There is no typographic resting point that signals "here is the action." The form card uses the identical `.card` styling as any other element would, making it visually indistinct from surrounding content.

### Pillar 2: Interaction & Animation Quality (3/4)

**Strengths:**
- Pixel-press button effect (lines 192-200): `translate(2px,2px)` + `box-shadow` reduction on hover, `translate(4px,4px)` on active. Tactile, on-brand, and correctly uses the CSS `transition` guard on `:disabled`.
- `fade-up` IntersectionObserver entrance (lines 1215-1219): clean, threshold 0.12 is appropriate, `observer.unobserve()` after trigger prevents re-firing.
- Score counter animation (lines 1056-1065): `setInterval` at 18ms with 40 steps is smooth. `Math.round()` prevents decimal display.
- Loading step cycle (lines 1139-1150): 3 rotating messages at 2200ms cadence provide perceived progress during API wait.
- `scrollIntoView` after result (line 1117): correctly guides focus to results without page reload.

**Issues:**

**Hero glow animation is low-signal.**
`emglow` keyframe (lines 621-624) pulses `text-shadow` on the `<em>` tag in h1 every 3.5s. On a dark background with a dark viewport, this is nearly imperceptible to users. The CPU cycle cost is nonzero and the value is near-zero. Recommend removing or replacing with a single static glow.

**Ticker hover-pause (line 693) is undiscoverable.**
`animation-play-state: paused` on hover is a desktop-only interaction with no affordance. Mobile users (primary target) cannot pause and read ticker content. Given that ticker content should be removed anyway (see Pillar 3), this is moot but worth noting as a pattern.

**Enter key handler fires on all inputs except email (line 1211-1213).**
`document.addEventListener('keydown', e => { if (e.key === 'Enter' && document.activeElement.id !== 'emailInput') runCheck(); })` — this means pressing Enter in the category select or keywords field also triggers the form submit. Correct behavior for product_name and product_url, but unexpected for select dropdown (user may press Enter to select an option, triggering a premature API call).

### Pillar 3: Trust Signals & Credibility (1/4)

**BLOCK — This pillar requires immediate action before launch.**

**Finding A: Fabricated live scan counter (lines 1222-1236)**

```javascript
const base = 180 + (new Date().getHours() * 7 + new Date().getDate() * 3) % 120;
let count = base;
// ...
setInterval(() => {
  if (Math.random() < 0.35) { count++; updateCount(); }
}, 8000);
```

This generates a fake "products scanned today" number derived from the current hour and date, then randomly increments it every 8 seconds. It is displayed in two places simultaneously:
- stat-row: "X products scanned today" (line 834)
- founding-bar: "· X scanned today" (line 792)

The number is not from a database. It is manufactured. This is fabricated social proof. It directly contradicts PBA rule 3-③ ("'안 하면 확실히 안 보인다' 메시지 (과장 금지)") and rule 8 ("과장 카피 금지"). CEO has already flagged this as dishonest.

**Finding B: Fabricated ticker with hardcoded product scores (lines 1239-1265)**

```javascript
const EXAMPLES = [
  { name: 'Ultimate Notion Planner', score: 23, cls: 's-low' },
  { name: 'Framer UI Kit Pro', score: 67, cls: 's-mid' },
  // ... 10 more hardcoded entries
];
```

These 12 product-score pairs are presented in a horizontally scrolling "live scan" ticker with no disclosure that they are examples. A creator reading "Framer UI Kit Pro — 67/100" has no way to know this is invented. This is materially misleading. The ticker's design (scrolling, live-dot in founding bar, "scanned today" counter) creates the false impression that these are real scans happening in real time.

**Finding C: Stat "23" and "3 in 5" with no sourcing (lines 826-831)**

```html
<span class="stat-num">23</span>
<div class="stat-label">avg score for digital products</div>

<span class="stat-num">3 in 5</span>
<div class="stat-label">creators invisible to AI right now</div>
```

These statistics have no footnote, source link, or qualifying "estimated" label. "3 in 5" especially reads as a research finding. If it is internal data from early scans, it must be attributed. If it is estimated, it must say "est." If it is fabricated, it must be removed.

**Finding D: "Founding creator spots open — first 200 get lifetime free reports" (line 792)**

The founding bar implies scarcity (200 spots) but has no counter showing remaining spots, no verification mechanism, and no indication this offer has an end date. If the 200 limit is real, it needs a remaining count. If it is soft urgency, it should be reworded to "early access" language without the numeric claim.

**What good trust signals look like here:**
- A real scan counter from Supabase with "beta data" label
- Actual testimonial quotes with creator names and product links
- "Methodology" link explaining what the score measures
- "Our data is live" footnote with a query timestamp on results

### Pillar 4: Content Clarity & Copy Strength (3/4)

**Strengths:**
- H1 "Does AI recommend your product?" (line 807): direct, specific, anxiety-triggering for the target user. Excellent.
- Sub-headline (line 808): "Find out in 10 seconds. Enter your product name and see your AI Visibility Score — the metric every digital creator needs to know." Clear, benefit-led, time-boxed.
- Hero tagline "IF AI DOESN'T KNOW YOU · YOU DON'T EXIST" (line 809): punchy, zero padding, works at the pixel font size.
- "How It Works" 3-step format (lines 869-890): clear sequential structure. "Takes 20 seconds" is concrete.
- Suggestion copy (lines 1026-1046): the actual improvement text is unusually specific and actionable ("Add an llms.txt file", "Create comparison content: 'Your Product vs [category alternatives]'"). This is a genuine differentiator and the strongest copy in the file.

**Issues:**

**Generic error message (line 1171):**
"Something went wrong. Please try again." — Standard placeholder. For a tool targeting non-technical creators, this provides no guidance. Better: "We couldn't reach the scanner. Check your connection and try again, or email hello@pickedby.ai."

**Email CTA micro-copy is flat (lines 984-985):**
"Get a weekly AI Visibility Report for your product — free. Track your score over time." Functional but passive. Compare to: "Know before your competitors do. Weekly AI Visibility Report — free." The benefit is the same but the frame is competitive urgency, which aligns with the product's core anxiety hook.

**Button label "SEND ▶" (line 988):**
At `font-size: 0.45rem` in Press Start 2P, this renders at ~7.2px. The button is functional but the label is nearly illegible on mobile. "GET REPORT" would be slightly more descriptive without adding length.

**"Category (optional)" and "Target audience (optional)" labels (lines 906, 924-925):**
The "(optional)" marker uses `color: #333` on dark background — effectively invisible (line 906: `<span style="color:#333;font-size:0.85em">(optional)</span>`). This creates uncertainty about whether these fields are required.

### Pillar 5: CTA Placement & Conversion Flow (2/4)

**The funnel structure as built:**

```
[1] Founding bar (sticky, scarcity claim)
[2] Header (logo + BETA badge)
[3] Hero (h1 + subtext + tagline)       ← first scroll position
[4] VS row (Google / ChatGPT / Claude / Perplexity)
[5] Stat row (3 fake statistics)
[6] Scan ticker (12 fake product scores)
[7] AI Vitals (3 feature cards)
[8] How It Works (3-step diagram)
[9] FORM CARD  ← primary conversion action
[10] Loading state
[11] Result (score + suggestions + email capture)
```

The form is the 9th content element (counting the header). On a 375px mobile screen, sections 3-8 occupy an estimated 1,800-2,200px of scroll distance before the form appears. The founding bar's scarcity message ("first 200") creates urgency at the top but then makes the user scroll through six more sections before they can act on that urgency.

**Specific issues:**

**No above-the-fold CTA.** The hero section (lines 804-810) ends at a tagline with zero interactive element. A user who lands, reads the H1, and wants to immediately try the tool has no affordance — they must scroll, hope, and discover the form.

**"How It Works" placement (lines 866-891) is backwards.** Explanation sections are for users who need convincing. Motivated users (who came from a social referral or ad) want to act first, understand second. The current order punishes the most motivated segment.

**Email capture is post-result only (lines 983-992).** Email is buried inside the result panel, shown only after a successful API scan. This means the entire email collection funnel depends on: (a) user finding the form, (b) user submitting the form, (c) API returning a result, (d) user reading the email prompt. Any failure in steps a-c means zero email collection. A secondary email capture above the form ("Get notified when full reports launch →") would collect emails from users who bounce before submitting.

**Recommended restructured flow:**
```
[1] Founding bar (sticky)
[2] Hero h1 + 2-line sub + FORM (collapsed: product name + URL + CTA)
[3] VS row (social proof for AI engines checked)
[4] How It Works (3 steps)
[5] AI Vitals (feature detail)
[6] Email capture strip (secondary, pre-result)
[7] Footer
```

### Pillar 6: Mobile Responsiveness (2/4)

**Single breakpoint at 480px (line 774):**

```css
@media (max-width: 480px) {
  .how-steps { grid-template-columns: 1fr; }
  .how-arrow { display: none; }
  .hero h1 { font-size: 1.75rem; }
  .score-num { font-size: 3rem; }
  .email-row { flex-direction: column; }
  .btn-email { padding: 0.75rem; }
  .vitals-grid { grid-template-columns: 1fr; }
  .vs-row { flex-direction: column; gap: 0.35rem; }
  .stat-row { grid-template-columns: 1fr; gap: 1px; }
}
```

Modern phones: iPhone 14 (390px width in portrait), Pixel 7 (412px), Galaxy S23 (393px) — all fall under the 480px threshold and receive the mobile layout correctly.

However, the 481-767px range — which includes iPhone 14 in landscape (844px, but common viewport), iPad mini portrait (768px edge case), and many Android tablets — receives the full desktop-density layout:
- `vitals-grid: repeat(3, 1fr)` at ~560px container = three ~170px columns, each with a 1.25rem emoji + 0.35rem label + 0.68rem description. Functional but dense.
- `stat-row: repeat(3, 1fr)` at 480-560px container = acceptable but tight.
- `how-steps: 1fr auto 1fr auto 1fr` — five-column grid with arrows at 481px container collapses to ~80px per step column. Arrow characters push into overflow territory.

**Container max-width is 560px (line 41).** This is actually mobile-first in spirit — the page was designed to be narrow. But the body padding is `0 1rem` (line 38) so on a 375px viewport the content area is 343px, which is correct. The breakpoint mismatch matters most in the 480-560px device range.

**Touch target sizes:**
- `.btn-primary` at `padding: 0.9rem 1rem` (line 179): ~44px height, correct.
- `.btn-email` at `padding: 0 1rem` (line 495-496): height is determined by the adjacent input's `0.7rem 0.85rem` padding = ~39px. Below 44px minimum.
- `.btn-share` at `padding: 0.78rem 1rem` (line 416): ~42px, borderline.
- `.btn-badge` at `padding: 0.75rem` (line 469): ~42px, borderline.

**Scroll behavior on result reveal (line 1117):**
`resultEl.scrollIntoView({ behavior: 'smooth', block: 'start' })` — on iOS Safari, smooth scroll behavior can conflict with the founding bar's `position: sticky`. The result card may scroll to a position where the founding bar overlaps the first 40px of the result card. No `scroll-margin-top` is set on `.result`.

**Placeholder color (line 160):**
`input::placeholder { color: #333; }` — `#333` on `#161616` (bg3) is approximately 2.0:1 contrast. WCAG requires 4.5:1 for text. Placeholders at this contrast are near-invisible, especially on OLED screens. Recommendation: `#555` minimum.
